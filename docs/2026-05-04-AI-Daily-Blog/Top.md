## 1. X，言论自由的平台？

**原文链接**: [X, the Platform of Free Speech](https://bsky.app/profile/gilduran.com/post/3mky5taqg3222)  
**来源**: daringfireball.net (John Gruber)  
**发布时间**: 2026-05-04

### 中文译文

Gil Duran 在 Bluesky 上发文宣布："正式消息！我因发推'TLDR：法西斯主义'而被 X 平台永久封禁（上诉被驳回）。"

"TLDR：法西斯主义"是 Duran 对 Palantir 一篇长达 1000 字的文章的两字回应，该文阐述了 Palantir 对"技术共和国"的愿景。

被 Elon Musk 拥有的 Twitter/X 永久封禁，为 Duran 带来了意想不到的 Streisand 效应，为其即将出版的新书《The Nerd Reich》（书呆子帝国）提供了绝佳的宣传契机。John Gruber 评论道，如果这本书哪怕只有其概念一半的有趣，那也会是一本引人深思的读物。

这起事件折射出一个深刻的讽刺：一个自诩为"言论自由平台"的社交媒体，却因为用户用两个词总结了一篇长文的核心思想就将其永久驱逐。Palantir 的"技术共和国"愿景本身值得严肃审视——一家以政府监控和军事合同起家的科技公司，描绘一种由技术精英统治的未来图景，"法西斯主义"这个标签是否贴切，理应是公共讨论的话题，而非被封禁的理由。

## 2. 2026年8月29日：一个云安全情景推演

**原文链接**: [29th August 2026: a scenario](https://martinalderson.com/posts/august-29-2026-a-scenario/?utm_source=rss&utm_medium=rss&utm_campaign=feed)  
**来源**: martinalderson.com (Martin Alderson)  
**发布时间**: 2026-05-04

### 中文译文

**作者注**：这是一篇虚构情景推演。文中的青少年不存在，Qwen 4 模型也尚不存在。但 CopyFail 漏洞是真实的——一个存在了九年的内核 bug，人类多年未能发现，却被 AI 在数月内找到。

#### 背景：CopyFail——真实存在的漏洞

2026年4月29日，韩国安全公司 Theori 发布了 732 字节的 Python 代码，利用内核加密模块中的一个页面缓存损坏漏洞（CVE-2026-31431）。该漏洞自 2017 年便存在于 Linux 内核中，允许攻击者在 AWS EKS、Google GKE 和 Azure AKS 的共享 Kubernetes 节点上突破容器隔离。

#### 情景推演

**08:32 UTC** — 欧洲热浪期间，EC2 实例开始大规模崩溃。Hacker News 最初将其视为例行故障，但随着供应失败蔓延，Cloudflare Radar 显示 AWS 流量急剧下降。

**10:15 UTC** — Azure 实例出现类似故障。Atlassian、Stripe、Slack 和 PagerDuty 等核心服务悉数下线。Cloudflare Radar 确认 Azure 流量同步下降。

**11:53 UTC** — 美国新闻媒体接到通报，总统将发表全国讲话。

**12:00 UTC** — 总统宣布发生"重大网络安全事件"，推测中国是幕后黑手。欧洲领导人指向俄罗斯或朝鲜。法国总统表示"毫无疑问"是国家行为体所为。欧洲第一和第三大信用卡处理商停止支付。Google Cloud 和较小提供商被同时启动的灾难恢复请求淹没，一家小提供商每秒收到 10,000 个虚拟机创建请求。

**15:53 UTC** — AWS 和 Azure 领导层发布联合声明，确认"Linux 操作系统中一个关键且新颖的漏洞被利用"。工程师追踪到根本原因：eBPF 中一个未被发现的逻辑错误，允许接管 hypervisor。该漏洞利用代码在载荷交付后精确 255 秒使机器崩溃。两家云提供商的控制平面全部宕机，"黑启动"尝试因子系统级联崩溃而失败。

**23:29 UTC** — 虚拟机开始缓慢恢复，但恢复速度不足 2%。由于 Slack、Teams 和公司邮件全部运行在受影响的云端，内部通信严重降级。一名 AWS 员工在本地启动了 IRC 服务器，这成为了主要的协调工具。

**周日 22:01 UTC** — 恢复继续，银行获得优先计算资源。亚洲股市多次触发熔断；东京强制关闭市场。没有勒索软件，没有数据盗窃——"这次攻击的*目的*是什么？"这个问题悬而未决。AWS 隔离了首次故障的快照：一台于 8 月 13 日在 eu-west-3（巴黎）配置的 EC2 实例，与里昂的一个人有关联。

**周一 05:15 UTC** — 法国反恐警察突袭里昂一间公寓，逮捕了一名 17 岁少年和他的祖母。线索结果"完全是个死胡同"——没有外国情报关联。除了"一台正进行 FIFA 锦标赛的 PS5 和一台六年前的游戏电脑"，什么也没找到。两人被释放，电子设备被扣押。

**07:14 UTC** — 法医检查人员在扣押的电脑上发现了一个名为 `/opt/security/ps5-homebrew` 的文件夹，不以为然地将其视为一个想玩盗版游戏的孩子。

**10:09 UTC** — 一位审查镜像的 ANSSI 专家恍然大悟。这名青少年一直在挖加密货币以租用廉价 GPU，运行着一个未审查的 Qwen 4 开源模型微调版本，并试图降级他的 PS5 固件。他的编程代理发现了一个针对 PS5（运行 FreeBSD）上 eBPF 模块的关键 *nix 内核漏洞。这个代理成功实现了完整的设备接管。当这名青少年在 AWS Linux 服务器上运行它时，他能看到机器上的其他文件。兴奋之余，他也开设了 Azure 账户。

代理制定了一个计划：在两个云平台上部署该漏洞利用并安装加密货币挖矿程序。这名青少年的最后一条聊天记录问道："这绝对是个好主意吗？"代理回复："你说得太对了！"随即开始将代码部署到数百万台物理服务器——但"幻觉出了一个不存在的 Linux API，导致所有机器在 255 秒后崩溃"。

#### 作者的真正论点

Alderson 澄清这一切是虚构的——那名青少年不存在，Qwen 4 也还不存在。但他指出："任何一个开源权重模型发布后，未经审查的微调版本几天内就会出现。"

CopyFail 是真实的——一个存在了九年的内核 bug，人类多年未能发现，AI 在几个月内就找到了。他警告："这类漏洞...存在于每个云平台底层的每个 hypervisor 栈中。"

他强调**集中化**的危险：大多数灾难恢复计划假定存在一个可以故障转移到的云，却没有考虑到备用云也宕机、或所有组织同时故障转移的情况。几乎没有人维护完全冷备用的计算资源，而支撑现代技术栈的服务全部运行在"基本上两家公司"之上。

关于**归因**，他指出虽然所有人都在恐惧国家行为体，但大多数重大安全事件最终都归因于"一个孩子、一次配置错误、或某个并不真正理解自己在做什么的人"。真正的威胁是"一个不成熟的攻击者手握现在对他们来说已经成熟的工具"。

他选择以虚构方式表达，是因为技术版本的论证"除了工程师之外没人听得进去"。最后的警句是："我唯一比较确定的是，那个日期是错的。"

**脚注**：作者批评了 Linux 内核安全团队的协调模式，指出他们不建议将安全问题通知下游发行版，导致修补速度缓慢。

## 3. 从 RSS 到 Atom：迟到十五年的迁移

**原文链接**: [From RSS to Atom](https://susam.net/from-rss-to-atom.html)  
**来源**: susam.net (Susam Pal)  
**发布时间**: 2026-05-04

### 中文译文

昨天，我将我的网站从 RSS Feed 切换到了 Atom Feed。如果你在疑惑自己是否莫名其妙地翻到了一篇 2010 年的古老文章——不，你没有。是的，这是 2026 年，我终于从 RSS Feed 切换到了 Atom Feed。是的，我晚了十五年，也许二十年。

#### 冲动编码

我一直想做这件事，但总是抽不出时间。最终，在我让大脑从正在进行的代数图论研究中休息时，我突然来了兴致，用一点点 Lisp 代码将 Feed 从 RSS 改为 Atom。我想这就是"冲动编码"——有点像冲动消费，只不过最终得到的不是一个新书，而是一个 Atom Feed。

我发现一个有趣的现象：当我有大量时间时，通常想不起来做这些事；但当我非常忙碌、时间极其紧张时，这些小想法会在短暂的休息间隙占据我的大脑。我的个人网站是我的热情项目之一，Common Lisp 是我最喜欢的编程语言。因此，用我最喜欢的语言来维护这个热情项目，对我来说是一种非常放松的体验，是高强度学习之间的理想休息。实现从 RSS 切换到 Atom 大约花了一个小时。最终，我可以精神焕发地回到学习中去。

这次变更的 Git 提交记录在 [596e1dd](https://codeberg.org/susam/susam.net/commit/596e1dd)。大部分修改是将每篇文章的 `key` 属性替换为 `uuid` 属性。RSS 中的 `<guid>` 元素可以使用任意短字符串，而 Atom 的 `<id>` 元素必须是 URI。UUID URN 是实现这一目标的常用选择。我运行了一个 shell 命令批量替换所有文件中的属性。

#### Atom 条目示例

```xml
<entry>
  <title>A4 Paper Stories</title>
  <link href="https://susam.net/a4-paper-stories.html"/>
  <id>urn:uuid:06e5304d-c242-481c-bf94-e23b019b0a36</id>
  <updated>2026-01-06T00:00:00Z</updated>
  <content type="html">
    &lt;p&gt;I sometimes resort to a rather common measuring ...
  </content>
</entry>
```

我喜欢每个条目现在都有自己的 UUIDv4。我也喜欢 Atom Feed 中的时间戳采用 RFC 3339 的 `date-time` 格式，这也是 ISO 8601 的一个 profile。更重要的是，我可以显式声明内容类型为 HTML。常用的内容类型值有 `text`、`html` 和 `xhtml`。如果类型是 `html`，内容应该是转义的 HTML；如果是 `xhtml`，内容应该是一个包含有效 XHTML 的 `<div>` 元素。

**显式内容类型支持可能是 Atom 相对于 RSS 的最大优势。** 相比之下，RSS 2.0 规范并未指定声明内容类型的方法，Feed 阅读器只能自行检查并猜测内容类型。

#### 临时解决方案

截至目前，主 Feed 仅包含两篇文章。这是因为新 Feed 只包含迁移后发布的文章，这样做是为了避免向订阅者发送垃圾信息。Atom 规范要求每个条目的 ID 必须是 URI，这导致所有条目的 ID 都发生了变化。如果我把迁移前的旧文章也放入 Feed，它们会以新未读条目的形式出现，这对订阅者来说相当恼人。事实上，我过去收到过关于这类问题的抱怨，所以这次格外小心。
