# Richard Dawkins 与 Claude 幻觉

**作者：Gary Marcus**

Richard Dawkins 在 2006 年出版了《上帝错觉》（The God Delusion），论证对上帝的信仰是一种错觉——一种在缺乏证据的情况下坚持的信念。今天，我认为我们在 AI 领域看到了一个类似的错觉：人们相信大语言模型具有它们实际并不具备的理解能力、意识或智能。

我称之为"Claude 幻觉"——但这不是 Claude 独有的问题，它适用于所有大语言模型。

Claude 幻觉的核心是：因为模型能够产生流畅、连贯、看似深思熟虑的文本，我们倾向于假设文本背后存在一个思考着的、理解着的心智。这是对语言输出与认知能力之间的范畴错误。

Dawkins 在谈到宗教时说："相信上帝存在的理由不比相信飞天面条怪物的理由更多。"我并非说相信 Claude 有意识同样荒谬——但结构上的相似性值得注意。在这两种情况下，人类都在将主体性投射到缺乏证据支持该投射的事物上。

当 Claude 写出一段优美的散文时，它不是在表达深刻的思想——它是在基于其训练数据中的统计模式预测下一个词元。这些统计模式恰好包含了大量人类深刻思想的痕迹，但它们不是思想本身。

这不是贬低 Claude 的技术成就。GPT-style 模型的能力确实令人印象深刻。但我们需要对它们到底是什么保持清醒——它们是语言模式匹配器，而不是有意识的存在。

# 一个为维护者设计的 GitHub

当前的 GitHub 是为消费者设计的——浏览代码的人、提交 issue 的人、以及在 README 下方的星星按钮。但它不是为维护者设计的。

维护者需要的不是更多的星星。他们需要更好的分诊工具。他们需要能减少噪音的通知系统。他们需要能区分"我应该立即关注的 issue"和"可以安全忽略的 issue"的方法。

"一个为维护者设计的 GitHub"应该是什么样的？

首先，默认的仓库视图不应该是 README，而是一个健康的仪表板。最近的 CI 运行状态、新 issue 的分类摘要、需要审查的 PR 队列、以及依赖更新。README 是给访客看的——维护者需要的是态势感知。

其次，通知系统需要从根本上重新设计。当前的通知模型将所有东西都塞进一个按时间排序的流中——issue 评论、PR 审查、release 通知、sponsor 消息。一个维护者合理关注的所有项目加起来，这个流很快就变得无法驾驭。

第三，也是最重要的——我们需要更好的工具来说"不"。现在，关闭一个 issue 或拒绝一个 PR 需要心理能量。每次你说不，你都在冒着与潜在贡献者产生摩擦的风险。一个为维护者设计的平台应该使友善地说不变得容易——也许是预设的模板，也许是关闭 issue 时自动添加的解释。

开源维护者正在经历精疲力竭的疫情。更好的工具不能解决所有问题，但它们确实能有所帮助。

# 编辑 LLM 辅助文章

**作者：Ibrahim Diallo**

我用 LLM 辅助写作已经有一段时间了。不是让 AI 替我写——而是用 AI 作为头脑风暴伙伴、结构顾问、以及初稿生成器。但有一件事我学到的教训很重要：AI 生成的初稿只是一个起点，真正的写作发生在编辑阶段。

编辑 LLM 辅助的文章与编辑自己的初稿有着微妙的不同。当你编辑自己的写作时，你知道你想说什么——你只是在调整表达方式。当你编辑 LLM 输出时，你需要首先判断它是否说对了——然后才能开始调整表达方式。

这意味着编辑 LLM 输出需要更多的"事实核查"阶段。LLM 倾向于自信地陈述错误的信息。它可能会编造统计数据、错误引用来源、或者流畅地写出听起来合理但实则不对的内容。作为人类编辑，你有责任捕捉这些错误。

另一个挑战是声音。LLM 的输出有一种特定的"风味"——过于正式、过度使用某些过渡词、避免鲜明的观点。让它听起来像你自己需要大量的重写。我不会让 LLM 生成的内容未经明显修改就出现在我的博客上。

但说到底，AI 辅助提高了我的产出质量。不是因为 AI 写得比我好——而是因为它让我能更快地通过"空白页"阶段，把更多的时间花在真正重要的事情上：把想法打磨成值得别人阅读的内容。
