# Bitwarden 如何加密和解密机密

**作者：Miguel Grinberg**

作为减少对大型科技公司依赖的一部分努力，我一直在研究如何自托管我的密码。在这个过程中，我深入研究了 Bitwarden 的加密架构。

Bitwarden 使用的是端到端加密模型。所有加密和解密操作都在客户端完成，服务器只存储加密后的密文。即使 Bitwarden 的服务器被攻破，攻击者也无法解密用户的密码库数据。

加密流程从用户的主密码开始。当你创建 Bitwarden 账户时，系统使用 PBKDF2 或 Argon2 密钥派生函数，通过你的主密码和电子邮件地址生成一个主密钥。这个主密钥永远不会发送到服务器——它只存在于你的设备上。

当你在密码库中添加一个密码项时，系统会生成一个随机的对称密钥来加密该项的数据。加密使用 AES-256-CBC 算法。加密后的数据连同初始化向量（IV）一起被上传到 Bitwarden 服务器。

解密过程相反：当你需要查看密码时，客户端从服务器下载加密的密码项，使用你的主密钥解密对称密钥，然后用该对称密钥解密密码项的内容。整个过程对用户透明，但安全模型非常优雅——服务器对你的数据一无所知。

对于团队或家庭共享的密码库，Bitwarden 使用公钥加密方案。每个成员有一对公私钥，共享密码项时使用接收者的公钥加密对称密钥，确保只有授权成员能够解密。

这种架构的核心优势在于：即使法律强制要求 Bitwarden 交出数据，他们也只能交出加密的密文，没有用户的密码根本无法解密。

# 集体速度不等于个体速度之和

**作者：Jim Nielsen**

我一直在思考关于速度的话题——特别是，速度并不利于智慧。

在组织环境中，有一个常见的误解：如果每个个体都工作得更快，整个组织就会变得更快。但实际情况是，集体速度并不是个体速度的简单相加。

想象一下高速公路。如果每辆车都以最大速度行驶，结果不是更快的交通，而是更多的交通事故和拥堵。同样，在软件开发中，如果每个开发者都在全速前进却不协调，结果往往是更多的技术债务、更频繁的合并冲突和更多的返工。

真正的集体效率来自于协调、沟通和在正确方向上的对齐。有时候，放慢脚步进行深思熟虑的规划，比每个人都急于完成各自的任务更有效。这不是懒惰，而是认识到复杂系统中的非线性效应。

当我看到那些推崇"更快、更多、更努力"的组织时，我不禁想起了"慢工出细活"的古老智慧。在一个每个人都跑得很快但没有方向的组织里，唯一确定的结果是——他们会更快地到达错误的目的地。

# 语音调制解调器：手机音频架构的怪异之处

**作者：Computer.rip**

如果你对现代手机有所了解，你可能已经注意到音频架构是多么的奇怪。

这一切的根源可以追溯到手机的起源——电路交换电话网络。在 2G GSM 时代，手机本质上就是一个无线电话机，语音通话是唯一的核心功能。整个音频子系统都是围绕实时、低延迟的语音传输而设计的。

随着 3G 和 4G 的到来，数据通信变得越来越重要，但音频架构的基础设计几乎没有改变。现代的 VoLTE（LTE 语音）和 VoNR（NR 语音）仍然继承了这个架构。语音信号经过独立的处理管线——从麦克风采集、音频编码器压缩、到基带芯片的专用语音通道——这些与数据管线完全分离。

更复杂的是，许多现代手机仍然保留着传统的电路交换音频模式作为后备方案。这意味着手机的操作系统必须同时支持两种完全不同的音频通路：一种是传统语音通话的窄带语音（8kHz 采样率），另一种是用于媒体播放的宽带音频（48kHz 甚至更高）。

这解释了为什么有时你在通话中切换到扬声器模式时，音质会有戏剧性的变化——系统正在不同的音频处理管线之间切换。它也是为什么 Android 的音频 HAL（硬件抽象层）是众所周知的复杂：它必须协调所有这些历史遗留的模式。

随着 5G 时代的到来，VoNR 承诺最终实现高质量的纯 IP 语音通话，但这段漫长的过渡期意味着我们可能在很长时间内都要与这些老旧的设计共存。
