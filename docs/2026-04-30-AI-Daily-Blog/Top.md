## 1. 英国 AISI 对 GPT-5.5 网络攻防能力的评估

**原文链接**: [UK AISI](https://www.aisi.gov.uk/blog/our-evaluation-of-openais-gpt-5-5-cyber-capabilities)

**来源**: UK AI Security Institute (英国人工智能安全研究所) via Simon Willison
**发布时间**: 2026-04-30

---

### 中文翻译

英国人工智能安全研究所（AISI）发布了其对 **OpenAI GPT-5.5 网络攻防能力**的正式评估报告。这是 AISI 继此前评估 Anthropic Claude Mythos 之后，对前沿 AI 模型安全能力的又一次系统性审查。

#### 评估框架

AISI 的评估聚焦于 GPT-5.5 在实际网络安全场景中的表现，包括：
- **CTF 挑战**（夺旗赛式安全挑战）
- **真实漏洞发现场景**
- **漏洞利用能力**
- **防御建议生成**

#### 核心结论

评估的核心发现是：**GPT-5.5 在漏洞发现方面的能力与人类网络安全专家相当**。

这一结论具有双重含义：
1. **积极面**：AI 可以成为防御者的有力工具，帮助识别和修复安全漏洞。
2. **隐忧面**：同样的能力也可能被恶意行为者利用，降低网络攻击的技术门槛。

#### 政策意义

AISI 强调需要为前沿 AI 模型的网络攻防能力建立**持续评估框架**。随着每一代新模型的发布，其网络能力都在提升，监管和安全评估必须跟上这一节奏。

> 此前，Simon Willison 也曾撰文分析过 AISI 对 Claude Mythos 的类似评估，指出这类独立第三方评估对于理解 AI 模型的安全边界至关重要。

## 2. Codex CLI 新增 /goal 命令：AI 代理的自我循环时代

**原文链接**: [GitHub Releases](https://github.com/openai/codex/releases/tag/rust-v0.128.0)

**来源**: OpenAI / Simon Willison
**发布时间**: 2026-04-30

---

### 中文翻译

OpenAI 的 **Codex CLI** 编码代理在 0.128.0 版本中引入了一项关键新功能：**`/goal` 命令**。

#### 什么是 /goal？

`/goal` 命令实现了社区此前讨论的 **Ralph 循环**模式（以提出该概念的开发者 Ralph 命名）。其工作方式如下：

1. **设定目标**：用户使用 `/goal` 命令设定一个编码目标（例如「为这个 API 添加完整的错误处理」）
2. **自主循环**：Codex 持续迭代——编写代码、运行测试、检查错误、修复问题
3. **条件终止**：循环持续到以下任一条件满足：
   - 目标被评估为已完成
   - 预设的 token 预算耗尽

#### 为什么重要？

这标志着 AI 编码代理从「单次指令-响应」模式向「**目标驱动-自主循环**」模式的关键演进：

| 旧模式 | 新模式 |
|--------|--------|
| 用户逐条下达指令 | 用户设定目标 |
| AI 执行一次即停止 | AI 循环执行直到完成 |
| 需要持续的人类监督 | 自主迭代 + 最终审查 |
| 适合简单、单步任务 | 适合复杂、多步骤任务 |

#### Simon Willison 的评价

Simon Willison 将此功能与 `ghuntley.com/ralph/` 上提出的 Ralph 循环概念相联系，指出 Codex 的实现允许用户定义自己的完成标准，是 AI 编码代理自主性的重大提升。

## 3. Zig 项目的反 AI 贡献政策：贡献者扑克与代码诚信

**原文链接**: [simonwillison.net](https://simonwillison.net/2026/Apr/30/zig-anti-ai/) | [Zig 行为准则](https://ziglang.org/code-of-conduct/)

**来源**: Zig Software Foundation / Simon Willison
**发布时间**: 2026-04-30

---

### 中文翻译

**Zig 编程语言项目**维护着开源社区中最为严格的反 LLM 政策之一。其行为准则明确规定：

> - 禁止在 Issue 中使用 LLM
> - 禁止在 Pull Request 中使用 LLM
> - 禁止在 Bug 跟踪器的评论中使用 LLM，包括翻译

#### 「贡献者扑克」：一种全新的开源哲学

Zig 软件基金会社区副主席 **Loris Cro** 撰写了一篇题为「贡献者扑克与 Zig 的 AI 禁令」的文章，Simon Willison 称其为「对全面禁止 LLM 辅助贡献的最佳阐述」。

核心理念颠覆了传统开源思维：

> **Zig 重视贡献者甚于他们的贡献。**

每个 PR 不是一段待合并的代码，而是一笔**对人的投资**——「审查和接受 PR 的主要目标不是引入新代码，而是帮助培养新的贡献者。」

即使一个 LLM 生成的 PR 在技术上完美无缺，它也未能满足项目的真正目标，因为「Zig 团队花在审查你工作上的时间，对帮助他们增加新的、自信的、值得信赖的贡献者没有任何帮助。」

#### 贡献者扑克（Contributor Poker）

Loris 用扑克牌打比方：「你玩的是人，不是牌。」在开源贡献中，**赌注是贡献者的长期潜力**，而不是他们第一个 PR 的内容。

Simon Willison 对此进行了延伸思考：

> 「如果 LLM 写了 PR 的大部分内容，为什么项目维护者要花时间审查和讨论那个 PR，而不是直接启动自己的 LLM 来解决同样的问题？」

这是一个深层的哲学问题：当代码生成变得廉价，人类审查和人际交流的价值反而变得更加珍贵。

#### Bun 分叉案例

一个现实的案例：**Bun**（JavaScript 运行时，2025 年 12 月被 Anthropic 收购）运营着自己的 Zig 分叉。他们通过并行语义分析和多代码生成单元，在 `bun compile` 上实现了 **4 倍性能提升**。然而，Bun 声明「目前不计划向上游提交，因为 Zig 严格禁止 LLM 创作的贡献。」

一位 Zig 核心贡献者还指出，即使抛开 LLM 问题，这个补丁本身也面临障碍——并行语义分析是一个规划已久的功能，对 Zig 语言本身有深远影响。

#### 数字气味

联合创始人 **Andrew Kelley** 在另一个场合解释称，使用 LLM 的贡献者带有一种可辨识的「**数字气味**」（digital smell）——人类错误与 LLM 幻觉有本质区别，前者源于疏忽或知识盲区，后者则是一种统计性的虚假模式匹眩，两者在代码审查中一眼可辨。
